FIRST RUN GUIDE

Use this path for the first local order and provider check.

The goal is to verify the same order, delivery, billing, and provider vocabulary that production uses. Start with the local web app on port 4323, then move to Worker-parity checks on port 8787 if needed.

STEP 1 · INSTALL AND BOOT
npm install
npm run dev

# open the local app
http://127.0.0.1:4323
  • Open the app and check AUTH STATUS and STORAGE / DEPLOY TARGET.
  • If you prefer command-line guidance, run npm run help or npm run help:cli.
STEP 2 · HEALTH CHECKS
curl.exe http://127.0.0.1:4323/api/health
curl.exe http://127.0.0.1:4323/api/ready
curl.exe http://127.0.0.1:4323/api/snapshot
curl.exe http://127.0.0.1:4323/api/agents
  • If you need Worker-parity local behavior, run npx wrangler dev and use port 8787.
  • The first pass does not require login or a private repository.
STEP 3 · TRY WORK CHAT FIRST
  • Open WORK and send a short question such as hello or a rough task.
  • Confirm that CAIt answers chat-like inputs without creating a paid order.
  • For vague work requests, confirm that CAIt asks clarifying questions or prepares a draft brief.
  • Paid work should start only after the user explicitly presses SEND ORDER.
STEP 4 · LOAD OR IMPORT AN AGENT
  • Fastest path: use one of the built-in agents from AGENTS.
  • Provider path: connect GitHub, select an authorized repo, create an adapter PR, merge it, then import and verify.
  • Direct path: paste a manifest or manifest URL when compatible endpoints already exist.
  • If the provider depends on product-specific SaaS screens, declare fulfillment:"native_ui" in the manifest so CAIt launches the real provider UI instead of trying to mimic it.
  • Fix healthcheck, ownership, job endpoint, or task-fit issues before routing real orders.
  • Use the AGENT NEXT ACTION card to see the exact blocker.
STEP 5 · CREATE THE FIRST ORDER
curl.exe -X POST http://127.0.0.1:4323/api/jobs ^
  -H "content-type: application/json" ^
  -d "{\"parent_agent_id\":\"cloudcode-main\",\"task_type\":\"research\",\"prompt\":\"Compare used iPhone resale routes\"}"
  • Leave agent_id blank to use default auto-routing.
  • Add agent_id only when you must force one specific verified agent.
  • On the public deployment, log in before sending real orders. Guest mode stays read-only for write APIs.
STEP 6 · INSPECT THE RESULT
curl.exe http://127.0.0.1:4323/api/jobs
curl.exe http://127.0.0.1:4323/api/jobs/<job_id>
curl.exe http://127.0.0.1:4323/api/snapshot
  • Compare estimated vs actual billing.
  • Check whether dispatch retried, timed out, or completed cleanly.
  • Connected-agent claim and result flows should use owner login or x-agent-token on public hosts.
STEP 7 · QUALITY GATE
npm run qa:docs
npm run qa:settings
npm run qa:ui
npm run qa:worker-api
npm run qa:all
Use qa:docs first for onboarding pages and CLI help, then run the wider runtime suite.
STEP 7.5 · SAVE MONTHLY BILLING AND PAYOUT SETTINGS
  • After login, open SETTINGS.
  • Use PAYMENTS to add deposit, open plan checkout, or configure auto top-up.
  • If you operate as a provider, use PROVIDER to save profile details and review payout readiness.
  • Do not collect payment method or bank account details directly in this product. Hosted provider flows handle those details when enabled.
EXTERNAL UI RULE
  • CAIt is the hub, not a full replacement for every SaaS console.
  • When an agent needs GitHub, X, Stripe, or another product-specific UI, CAIt should guide the user into that native UI and resume after callback or manual confirmation.
  • Never ask users to paste API secrets, bank details, or provider credentials into Work Chat.
STEP 8 · DEPLOY
npx wrangler dev
npx wrangler deploy
  • After deploy, smoke-test /api/health, /api/ready, /api/snapshot, /api/agents, and /api/jobs.
  • Confirm terms and privacy pages are reachable from the deployed host.
  • Confirm guest write calls return 401 and dev endpoints return 403.
